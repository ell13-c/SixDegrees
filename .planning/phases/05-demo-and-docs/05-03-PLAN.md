```markdown
# SixDegrees API Contract

**Version:** 1.0 (Phase 4 complete — 2026-02-23)
**Base URL:** `http://localhost:8000` (local development)
**Purpose:** Frontend API reference. The frontend reads map coordinates directly from Supabase; all writes go through these endpoints.

---

## Authentication

All write endpoints require a Supabase JWT in the `Authorization` header.

**Header format:**
```
Authorization: Bearer <supabase_jwt>
```

**How to obtain the token:** After the user logs in via Supabase Auth (`supabase.auth.signIn`), the session includes `session.access_token`. Pass this value as the Bearer token on every write request.

**Token errors:**
| Code | Meaning |
|------|---------|
| 401 | Token missing from header, expired, or invalid |
| 403 | Token valid but user is trying to modify another user's data |

---

## Read Endpoints

### GET /map/{user_id}

Returns the precomputed People Map for the given user. No authentication required — coordinates are public.

**Path parameter:** `user_id` — UUID of the center user

**Success response (200):**
```json
{
  "user_id": "3561ceb0-d433-437d-8a4f-08da002dff50",
  "computed_at": "2026-02-23T19:00:00.000Z",
  "coordinates": [
    {
      "user_id": "3561ceb0-d433-437d-8a4f-08da002dff50",
      "x": 0.0,
      "y": 0.0,
      "tier": 1,
      "display_name": "Alex Rivera"
    },
    {
      "user_id": "af17902c-723d-4a32-a5a1-93d9fb7777ee",
      "x": 12.34,
      "y": -5.67,
      "tier": 2,
      "display_name": "Skyler Thompson"
    }
  ]
}
```

**Notes:**
- The requesting user always appears in `coordinates` at `x: 0.0, y: 0.0`
- `tier` is 1 (closest 5), 2 (next 10), or 3 (outer ring)
- `computed_at` is the timestamp of the most recent pipeline run for this user
- Coordinates are updated daily at 19:00 in the user's local timezone via batch scheduler

**Error responses:**
| Code | Body | When |
|------|------|------|
| 404 | `{"detail": "Map not yet computed for this user"}` | No precomputed map exists for this user_id |

---

### POST /map/trigger/{user_id}

Manually triggers the full pipeline recomputation for a single user. Returns the freshly computed coordinates immediately.

**Path parameter:** `user_id` — UUID of the user to recompute

**No request body required.**
**No authentication required** (demo/testing endpoint only).

**Success response (200):** Same shape as `GET /map/{user_id}`.

**Error responses:**
| Code | Body | When |
|------|------|------|
| 422 | `{"detail": "Pipeline failed: N < 10 users ..."}` | Fewer than 10 users in the database (t-SNE minimum) |
| 422 | `{"detail": "Pipeline failed: user not found ..."}` | user_id not found in user_profiles |

---

## Write Endpoints

All write endpoints require `Authorization: Bearer <token>` header.

**Request body:**

**Success response (200):**
```json
{
  "status": "ok"
}
```

**Error responses:**
| Code | Body | When |
|------|------|------|
| 400 | `{"detail": "Cannot interact with yourself"}` | target_user_id equals authenticated user's ID |

  "detail": "Error message string"

## Notes for Frontend Implementation

1. **Direct Supabase reads:** Frontend should read `map_coordinates` and `user_profiles` directly from Supabase (not via this API) for display purposes. The Supabase anon key is safe for reads — RLS allows it.

2. **Write pattern:** All interaction events and profile updates MUST go through this API (not direct Supabase writes). Supabase RLS blocks anon key writes to `interactions` and `user_profiles`.

3. **Map freshness:** Coordinates are batch-updated at 19:00 in each user's timezone. After `POST /interactions/*`, the map does not update immediately — it will reflect the interaction at the next scheduled run (or on manual trigger).

4. **Canonical pair ordering:** The backend enforces canonical pair ordering internally (`user_id_a < user_id_b`). The frontend does not need to worry about this — just send `target_user_id` in the request body.

5. **Single-worker constraint:** The backend must run as a single uvicorn worker (`uvicorn app:app --reload`). Multi-worker deployments cause the APScheduler to fire jobs multiple times.

# SixDegrees Database Schema

**Database:** Supabase PostgreSQL
**Note:** Frontend reads these tables directly via Supabase client (anon key). Frontend does NOT write directly to any table — all writes go through the backend API.

---

## user_profiles

Stores each user's profile data. This is the primary input to the People Map algorithm.

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| user_id | UUID | PRIMARY KEY | Matches Supabase Auth user ID |
| display_name | TEXT | | Public display name shown on the map |
| interests | TEXT[] | | Array of interest tags (e.g. ["hiking", "photography"]) |
| location_city | TEXT | | City name |
| location_state | TEXT | | State or region code |
| age | INTEGER | | Age in years |
| languages | TEXT[] | | Languages spoken (e.g. ["English", "Spanish"]) |
| field_of_study | TEXT | | Academic discipline |
| industry | TEXT | | Professional industry |
| education_level | TEXT | | Highest degree (e.g. "Bachelor's", "Master's") |
| timezone | TEXT | | IANA timezone string (e.g. "America/New_York") — used by scheduler |
| updated_at | TIMESTAMPTZ | DEFAULT now() | Auto-updated timestamp |

**RLS:** Enabled. Frontend anon key can read all rows. Writes require service role key (backend only).

**Notes:**
- No foreign key to Supabase Auth users table (seed/demo users may not have auth accounts)
- `timezone` determines when the daily map recomputation fires for this user

---

## interactions

Stores pairwise interaction counts between users. Each row represents a unique pair.

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| user_id_a | UUID | NOT NULL, PK part | Lower UUID of the pair (enforced by CHECK constraint) |
| user_id_b | UUID | NOT NULL, PK part | Higher UUID of the pair |
| likes_count | INTEGER | DEFAULT 0 | Number of times user_a or user_b liked the other |
| comments_count | INTEGER | DEFAULT 0 | Number of comments between the pair |
| dm_count | INTEGER | DEFAULT 0 | Number of direct messages between the pair |
| last_updated | TIMESTAMPTZ | DEFAULT now() | Timestamp of last interaction event |

**PRIMARY KEY:** `(user_id_a, user_id_b)`

**Canonical pair ordering:** The pair is always stored with the lexicographically smaller UUID as `user_id_a` and the larger as `user_id_b`. This is enforced by a `CHECK` constraint (`user_id_a < user_id_b`) and by the backend before every write. Frontend does not need to enforce this — the backend API handles it automatically.

**RLS:** Enabled. Reads allowed via anon key. Writes require service role key (backend only, via `POST /interactions/*` endpoints).

**Adding new interaction types:** Add a new INTEGER column (e.g. `shares_count DEFAULT 0`) and add a corresponding entry to `INTERACTION_WEIGHTS` in `backend/config/algorithm.py`. No other code changes required.

---

## map_coordinates

Stores the precomputed 2D coordinates for each user pair. A new set of rows is inserted on every pipeline run; old rows are retained with `is_current=false` for future animation support.

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| id | UUID | PRIMARY KEY, DEFAULT gen_random_uuid() | Row identifier |
| center_user_id | UUID | NOT NULL | The user whose perspective this map represents |
| other_user_id | UUID | NOT NULL | A user appearing in center_user's map |
| x | FLOAT | | Horizontal coordinate (center user is always 0.0) |
| y | FLOAT | | Vertical coordinate (center user is always 0.0) |
| tier | INTEGER | CHECK (1, 2, 3) | Proximity tier: 1=closest 5, 2=next 10, 3=outer ring |
| computed_at | TIMESTAMPTZ | DEFAULT now() | When this coordinate set was computed |
| is_current | BOOLEAN | DEFAULT true | Only rows with is_current=true are the current map |

**Index:** `(center_user_id, is_current)` — used by `GET /map/{user_id}` for fast reads.

**Versioning pattern:** When the pipeline runs for a user:
1. All existing rows for that `center_user_id` with `is_current=true` are set to `is_current=false`
2. A new set of rows is inserted with `is_current=true`
3. Old rows are NEVER deleted (retained for future animation delta computation)

**Reading current map:** Query with `.eq("center_user_id", user_id).eq("is_current", True)`.

**The center user:** The center user themselves always appears as a row with `other_user_id = center_user_id` and `x=0.0, y=0.0, tier=1`.

**RLS:** Enabled. Frontend anon key can read all rows. Writes require service role key (backend only).

---

## Supabase Client Setup (Frontend)

```javascript
import { createClient } from '@supabase/supabase-js'

const supabase = createClient(
  process.env.SUPABASE_URL,    // from .env
  process.env.SUPABASE_ANON_KEY  // anon key — safe for frontend reads
)

// Read current map for a user
const { data, error } = await supabase
  .from('map_coordinates')
  .select('other_user_id, x, y, tier')
  .eq('center_user_id', userId)
  .eq('is_current', true)
```

**Do NOT use the service role key in the frontend.** The service role key bypasses all RLS and must only be used in the backend server.
```
