## Integration Check Summary

**Checked by:** gsd-integration-checker
**Wired connections:** 44/46 requirement chains verified end-to-end
**Issues found:** 2 low-severity spec/doc mismatches, 1 architectural note

### Issues Found

#### ISSUE-1: `/interactions/dm` response detail string mismatch (Low)
- **Location:** `backend/routes/interactions.py:38`
- **Code:** `column.replace('_count', '')` on `"dm_count"` → produces `"dm recorded"`
- **Docs:** `docs/API_CONTRACT.md` specifies `{"detail": "dms recorded"}`
- **Impact:** Functional behavior is correct; response string differs from spec by one character (`"dm"` vs `"dms"`)
- **Affected:** WRITE-03, SPEC-01

#### ISSUE-4: `DB_SCHEMA.md` wrong env var name (Low)
- **Location:** `docs/DB_SCHEMA.md:124`
- **Doc says:** `SUPABASE_SERVICE_ROLE_KEY`
- **Code uses:** `SUPABASE_KEY` (in `backend/config/supabase.py:17`)
- **Impact:** A developer following only `DB_SCHEMA.md` would use the wrong env var name and get a startup failure
- **Affected:** SPEC-02

#### ISSUE-2: Sync job on AsyncIOScheduler (Informational)
- **Location:** `backend/services/map_pipeline/scheduler.py:27`
- **Detail:** `_run_pipeline_for_timezone` is a synchronous `def` registered on `AsyncIOScheduler`. APScheduler 3.x runs sync callables in a thread pool executor — no event loop blocking, no correctness failure. Under load, the batch run occupies a thread for its entire duration.
- **Status:** Not a gap; architecturally acceptable for this milestone's scale (20 users)
- **Affected:** SCHED-01 (informational only)
